01. INTRODUCTION

Multi-Node Blockchain Simulation

This program represents a **decentralized blockchain network** operating inside isolated Docker containers.

To conceptualize the system, think of the network as **5 independent post offices (nodes)**, where each maintains its own physical copy of a **ledger book (the blockchain)**.

  • Consensus Rules are the strict guidelines preventing anyone from forging records.
  • Socket Networking acts as the messaging network immediately broadcasting new transfers or pages.
  • Smart Contracts are self-executing agreements running locally on each post office's desk.

config.py

Loads parameters like mining difficulty, reward token amounts, and socket network ports.

app.py

Bridges user HTTP requests with the core consensus and socket network engines.

blockchain/

Contains core blockchain components (blocks, transactions, networking, consensus).

static/

Web Dashboard client assets (HTML, CSS, JS) allowing active browser control of the network.

02. INITIALIZATION

Container Booting & Disk Load

The entire network runs on virtualized nodes. The system initializes state using the following workflow:

  • Docker Compose Orchestration: Spins up 5 instances concurrently. `node1` acts as bootstrap; `node2-5` rely on config variables to connect.
  • Persistent Storage loading: Each node checks if `blockchain.json` exists in its `/data` volume. If present, the history is restored.
  • Genesis Block Generation: If no history exists, nodes generate Block #0 (Genesis Block) using a static timestamp `1700000000.0`. This ensures all nodes compute the exact same starting block hash.

03. NETWORKING

Dynamic P2P Node Discovery

Nodes communicate over direct TCP sockets using custom length-prefixed JSON serialization.

  • Server Thread: Each node listens on socket port `6000` for peer updates.
  • Bootstrap Registry: Non-bootstrap nodes connect to `node1:6000` and send a `register` message.
  • Peer Sharing: `node1` stores their details and replies with its entire list of known peers.
  • Mesh Broadcasting: `node1` broadcasts `new_peer` to the rest of the network, enabling nodes to build a connected mesh.

04. MEMPOOL & HASHING

Transaction Creation & Propagation

Transactions contain `sender`, `recipient`, `amount`, `timestamp`, and optional `data` fields.

  • Uniqueness Hashing: The unique `transaction_id` is computed using SHA-256 over all parameters. Any attempt to modify a value results in a completely different ID.
  • Mempool Buffering: Added transactions are buffered in the local `pending_transactions` list (thread-safe mempool).
  • Network Flood Fill: The receiving node propagates the transaction to all active peers, which validate it and propagate it further.

05. VIRTUAL MACHINE

Dynamic Smart Contract Engine

Smart contracts do not require a separate persistent state database. Instead, they run dynamically via `ContractVM` processing.

  • Deployment: Triggered when a transaction is mined targeting `"contract_deploy"`, specifying contract name, type, and parameters.
  • Replays: To check states, the VM starts at block 0 and chronologically "replays" every transaction associated with the contract address.
  • Types: Supports three built-in types: **Counter** (count integers), **Token** (mint/transfer ERC-20 style assets), and **Guestbook** (message logs).

06. CONSENSUS ENGINE

Proof-of-Work Mining Loop

Mining groups pending transactions, issues coin rewards, and computes cryptographic proof.

  • Block Reward: Node automatically inserts a reward transaction (`10.0` CTK to itself) before starting.
  • Non-blocking loop: The node runs the PoW CPU cycle outside the database lock to keep other API requests responsive.
  • Proof-of-Work puzzle: The miner loops the `proof` nonce until the block's SHA-256 hash contains the necessary leading zeros (difficulty level).
  • Stale check: On completion, the node locks the database and checks if the chain progressed while it was mining. If so, the block is discarded.

07. CONFLICT RESOLUTION

Synchronization & Longest Chain Rule

If blocks are mined simultaneously or network latency cuts off a node, chains will diverge. Consensus resolves this:

  • Validation checks: Incoming blocks must verify: `index == last_index + 1`, correct proof-of-work, and correct previous hash pointer.
  • Fork detection: If an incoming block has an index greater than `last_index + 1`, the node knows it missed blocks.
  • Get Chain request: The out-of-sync node triggers a `get_chain` command to request the full ledger sequence from its peer.
  • Longest Chain Rule: If the peer's chain is valid and longer, the local chain is replaced completely, and mempool transactions are synchronized.

08. WEB INTERFACE

Visual Control Dashboard

The system includes a visual dashboard built in vanilla web components allowing active system operations:

  • Node Selector Grid: Pick any of the 5 nodes to inspect its state, ports, roles, and connected peers.
  • Transaction and Contract Forms: Construct wallet transfers, deploy smart contracts, and send action payloads to contracts.
  • Telemetry Explorer: Renders block history chronologically, displaying transactions, proofs, and transaction hashes.
  • Live Mempool Display: Tracks transactions buffered in the mempool waiting to be mined.
